STI_dac_v1.2: ZYNQ-7020 ELECTRICAL STIMULATION FIRMWARE
======================================================

1. OVERVIEW
-----------

This directory contains the FPGA hardware design and ARM firmware for the
custom wearable, reconfigurable electrical stimulation system described in
the associated study.

The design was developed with Xilinx Vivado Design Suite 2018.3 for a
Zynq-7000 XC7Z020 device. It combines:

  * Processing System (PS): bare-metal software running on the ARM
    Cortex-A9 processor; and
  * Programmable Logic (PL): deterministic stimulation timing, DAC control,
    electrode-switch control, and output-path sequencing.

The repository is supplied to allow reviewers to inspect the implementation
and reproduce the FPGA bitstream and ARM executable. The custom stimulation
PCB and its peripheral devices are required for full hardware reproduction.


2. DEVELOPMENT ENVIRONMENT
--------------------------

Required software:

  * Xilinx Vivado Design Suite 2018.3 (64-bit)
  * Xilinx SDK 2018.3

Target device:

  * Xilinx Zynq-7000 XC7Z020
  * Exact Vivado part: xc7z020clg400-2
  * ARM processor: Zynq PS7 Cortex-A9
  * FPGA fabric clock used by the custom stimulation IP: 50 MHz

Using the original 2018.3 toolchain is recommended. Opening the project in a
newer Vivado/Vitis release may upgrade the block design or IP and can change
generated files.

4. SYSTEM ARCHITECTURE
----------------------

The control path is:

  Host/mobile application
        |
        | BLE module operating as a UART bridge
        v
  Zynq PS UART0 at 115200 baud
        |
        | 16-byte stimulation command
        v
  ARM Cortex-A9 bare-metal firmware
        |
        | four 32-bit writes over AXI4-Lite
        v
  Custom sti_driver IP in programmable logic
        |
        +-- stimulation timing and left/right path control
        +-- DAC serial data and synchronization
        +-- four electrode-switch serial-data streams
        +-- latch, clear, chopping, balancing, and power-control signals

The Zynq PS master accesses the custom AXI4-Lite peripheral at:

  Base address: 0x43C00000
  Address range: 64 KiB

The custom interface contains four 32-bit registers at offsets 0x00, 0x04,
0x08, and 0x0C.


5. PROCESSING-SYSTEM (ARM) RESPONSIBILITIES
-------------------------------------------

The bare-metal application in STI_dac_v1.2.sdk/user_sti/src/main.c performs
the following functions:

  1. Configures PS UART0 for normal operation at 115200 baud.
  2. Receives stimulation commands with an interrupt-driven UART handler.
  3. Detects the two-byte command header 0x53, 0x74.
  4. Collects a complete 16-byte frame.
  5. Packs the frame into four 32-bit words and writes the words to the
     custom PL peripheral through AXI4-Lite.
  6. Controls the BLE module through four EMIO GPIO signals:
       EMIO 54: link-status input
       EMIO 55: reset output
       EMIO 56: command/data-mode output
       EMIO 57: low-power-mode output
  7. Configures PS I2C1 at 100 kHz and initializes an MPU6050 inertial sensor
     at 7-bit address 0x68.
  8. Samples MPU6050 gyroscope data from a 20 ms private-timer interrupt
     (nominally 50 Hz).
  9. Samples the Zynq XADC VP/VN input and converts it to the monitored
     battery voltage using the board divider factor 46/15.
 10. Prints gyroscope and battery-monitoring information through UART.

The firmware is a standalone/bare-metal application; no Linux operating
system is used.


6. PROGRAMMABLE-LOGIC RESPONSIBILITIES
--------------------------------------

The stimulation_ctrl module in sti_pl_dri.v operates from the 50 MHz AXI
clock (20 ns period). Its functions include:

  * loading stimulation parameters from the four AXI registers;
  * generating the stimulation repetition period;
  * generating left-side and right-side pulse windows;
  * controlling the output-power and stimulation-enable signals;
  * converting the current control byte into the serial DAC word;
  * generating serial clock/data/latch signals for four 16-channel
    electrode-switch devices;
  * controlling the analog multiplexer selection;
  * generating chopping and load/balancing intervals after active pulses;
  * clearing the electrode switches when stimulation is disabled.

The electrode configuration comprises four independent 16-bit masks, for a
total of 64 switch-control bits. These masks are shifted simultaneously on
DIN1-DIN4 and latched through LE_n.


7. COMMAND FORMAT AND AXI REGISTER MAP
--------------------------------------

The UART command is 16 bytes long:

  Byte 0       Header 0: 0x53
  Byte 1       Header 1: 0x74
  Byte 2       Stimulation frequency control byte
  Byte 3       Current/DAC control byte
  Bytes 4-5    Electrode mask
  Bytes 6-7    Electrode mask
  Bytes 8-9    Electrode mask
  Bytes 10-11  Electrode mask
  Byte 12      Mode and enable flags
  Byte 13      Pulse-width control byte
  Bytes 14-15  Reserved in the current implementation

The ARM application packs these bytes into the AXI registers as follows:

  Register 0, offset 0x00
    Bits 31:24  Frequency byte (command byte 2)
    Bits 23:16  Current/DAC byte (command byte 3)
    Bits 15:8   Header byte 0
    Bits 7:0    Header byte 1

  Register 1, offset 0x04
    Bits 31:16  Electrode mask from command bytes 4-5
    Bits 15:0   Electrode mask from command bytes 6-7

  Register 2, offset 0x08
    Bits 31:16  Electrode mask from command bytes 8-9
    Bits 15:0   Electrode mask from command bytes 10-11

  Register 3, offset 0x0C
    Bits 31:16  Reserved command bytes 14-15
    Bits 11:10  Stimulation mode
    Bit 9       Output-power switch
    Bit 8       Stimulation enable
    Bits 7:0    Pulse-width control byte

The current implementation interprets the frequency byte as a frequency in
hertz when calculating the PL repetition period. A nonzero frequency must be
provided. One pulse-width count corresponds to 2 microseconds in the PL
timing logic. The current byte is loaded into the DAC data field. Conversion
from this digital control value to physical output current depends on the
external DAC, analog circuitry, load, and experimental calibration.


8. STIMULATION SEQUENCE
-----------------------

For each stimulation period, the PL divides the active sequence into
intervals derived from the programmed pulse width:

  * electrode-switch and DAC control data are loaded;
  * the optional left path is activated during its assigned pulse window;
  * a chopping/load-balancing interval follows the left pulse;
  * the optional right path is activated during its assigned pulse window;
  * a second chopping/load-balancing interval follows the right pulse;
  * outputs remain inactive for the rest of the repetition period.

The two mode bits independently enable the left and right pulse windows.
Exact pin-level behavior can be reviewed in the always blocks of
sti_pl_dri.v.


9. BUILDING THE FPGA HARDWARE
-----------------------------

1. Install and launch Vivado 2018.3.
2. Open STI_dac_v1.2.xpr.
3. Open the system block design and allow Vivado to validate or regenerate
   output products if requested.
4. Confirm that the selected part is xc7z020clg400-2.
5. Run synthesis.
6. Run implementation.
7. Generate the bitstream.
8. Export the hardware to SDK and include the generated bitstream.

The implemented project already contains system_wrapper.bit and build
reports. The stored routed timing report indicates that the analyzed timing
paths met their constraints. The placed utilization report records
approximately 1,087 slice LUTs and 784 slice registers for this build.


10. BUILDING THE ARM APPLICATION
--------------------------------

1. Launch Xilinx SDK 2018.3 from Vivado after exporting the hardware, or
   open the STI_dac_v1.2.sdk workspace directly.
2. Confirm that system_wrapper_hw_platform_0 is the active hardware
   platform.
3. Regenerate user_sti_bsp if SDK reports stale BSP files.
4. Build the FSBL project.
5. Build the user_sti application in the Debug configuration.
6. Program the FPGA and launch user_sti.elf on ps7_cortexa9_0 for a JTAG
   test, or generate a boot image for nonvolatile boot.

The application links against the Xilinx standalone BSP and uses the XUartPs,
XScuGic, XGpioPs, XIicPs, XScuTimer, XAdcPs, and custom sti_driver drivers.


11. GENERATING AND USING BOOT.bin
---------------------------------

A pre-generated image is supplied at:

  STI_dac_v1.2.sdk/user_sti/bootimage/BOOT.bin

The image contains, in order:

  1. FSBL/Debug/FSBL.elf
  2. system_wrapper_hw_platform_0/system_wrapper.bit
  3. user_sti/Debug/user_sti.elf

The supplied user_sti.bif records absolute paths from the original
development computer. Before regenerating BOOT.bin on another computer,
update those paths in SDK's Create Boot Image dialog or create a local BIF
that points to the corresponding files in the current workspace.

The correct board boot-mode selection and programming procedure depend on
the custom PCB and its nonvolatile-memory configuration. QSPI is enabled in
the Zynq PS configuration.


12. HARDWARE INTERFACES
-----------------------

The XDC file maps the custom stimulation interface to XC7Z020 package pins
using LVCMOS33. Principal signal groups are:

  * DAC: sync_n, da_sdin, vdd, and SCLK;
  * electrode switches: CCLK, LE_n, DIN1-DIN4, DOUT1-DOUT4,
    and CLR1-CLR4;
  * stimulation paths: sti_L, sti_R, loadL_n, loadR_n, cut_wave,
    sti_on, and vcc_ctrl;
  * analog selection: acs[2:0];
  * BLE control: GPIO_bluetooth_tri_io[3:0];
  * MPU6050 I2C: IIC_1_0_scl_io and IIC_1_0_sda_io.

These assignments are specific to the custom stimulation PCB. They must not
be reused on another board without checking the schematic, I/O-bank voltage,
pin compatibility, and external circuit behavior.


13. REPRODUCIBILITY NOTES
-------------------------

  * The repository includes the Vivado project, block design, constraints,
    custom HDL, SDK application, BSP, FSBL, bitstream, ELF files, reports,
    and a pre-generated boot image.

    they can contain cached metadata and original absolute paths.
  * Full functional reproduction requires the matching custom stimulation
    PCB, BLE/UART module, MPU6050 connection, analog front end, electrode

    included in this version.
  * The pre-generated outputs document one completed build; rebuilding with
    another tool version may produce different implementation results.


14. SAFETY AND INTENDED USE
---------------------------

This is research firmware, not certified medical-device software. It must
not be used for clinical diagnosis or treatment.

Electrical stimulation can cause injury if the hardware, electrodes,
waveform, current, pulse width, frequency, duty cycle, placement, or
operating procedure is inappropriate. The firmware does not by itself
establish safe physical output limits. Safe operation depends on the custom
analog hardware, calibrated transfer function, firmware inputs, independent
protection mechanisms, approved experimental protocol, ethics approval, and
qualified supervision.

Reviewers without the matching hardware should inspect or rebuild the code
without connecting the stimulation output to a person or animal.
